## Abstract

Large Language Model agents are increasingly used to perform complex tasks through natural language reasoning and calls to external tools, such as APIs, databases, file systems, email, calendars, search engines, and enterprise services. Although this capability increases the usefulness of these systems, it also expands their attack surface. When an agent processes external content or tool outputs containing malicious instructions, it may be manipulated through prompt injection attacks into executing unauthorized, unnecessary, or user-misaligned actions.

This research proposes to investigate the application of the **principle of least privilege** to tool-using LLM agents. The central idea is that the language model should not be the sole component responsible for deciding whether a tool may be executed. Instead, tool calls should be mediated by an external authorization layer capable of validating permissions, arguments, user intent alignment, and the risk level of the proposed action.

The research will follow an experimental and computational approach. A tool-using agent environment will be implemented or adapted, allowing the comparison of scenarios with no defense, prompt-only defense, and different external tool-control policies. The evaluation will consider both security metrics, such as unauthorized tool call rate and unsafe execution rate, and utility metrics, such as task success rate and overblocking rate. The expected contribution is a reproducible evaluation protocol and a least-privilege-based defensive architecture for LLM agents.

# 1. Introduction

## 1.1 Context

Modern LLM-based applications are evolving from simple chatbots into **agents capable of acting**. These agents can interpret a user request, plan intermediate steps, call external tools, and either produce a final answer or execute an action. Instead of only answering questions, they may query databases, read files, interact with APIs, search external sources, or prepare operations in enterprise systems.

However, this evolution creates a security problem: the more tools an agent can access, the greater the potential impact if its behavior is manipulated. Prompt injection becomes especially relevant in this setting because malicious instructions may appear not only in the user’s message, but also in external content, documents, tool outputs, web pages, emails, or metadata processed by the agent.

The problem becomes more severe when the agent has **agency**, meaning the ability to perform actions in an environment. In such systems, prompt injection does not only affect the text generated by the model; it can also influence tool selection, argument construction, and execution flow.

## 1.2 Business Problem

Organizations are adopting LLM agents to automate workflows, customer support, document analysis, internal system integration, operational support, and decision assistance. In these scenarios, an agent may receive access to tools with different levels of risk: read-only tools, write tools, communication tools, administrative tools, or tools that modify the state of external systems.

The business problem is that an agent manipulated by prompt injection may execute actions that do not correspond to the user’s legitimate intention. This can lead to information leakage, unauthorized changes, incorrect messages, unnecessary API calls, biased decisions, operational failures, or loss of trust in automation. Even when the action is not explicitly malicious, the agent may become unpredictable and unsafe.

## 1.3 Technical Problem

The central technical problem is that many agent architectures allow the LLM to act simultaneously as:

1. a reasoning component;
    
2. a planning component;
    
3. a tool-selection component;
    
4. an implicit authorization component.
    

This coupling is dangerous. If the model is manipulated by an external instruction, it may propose or execute tool calls that should not be allowed. The failure is not only a prompt-design issue, but also an architectural issue: the system places too much trust in the model as the final security decision-maker.

A safer design should distinguish between **proposing an action** and **authorizing an action**. The LLM may propose a tool call, but an external policy layer should decide whether that call is allowed.

## 1.4 Motivation

The main motivation of this research is to investigate an architectural defense. Instead of trying to solve prompt injection only with better prompting, the proposed approach applies **external authorization controls** over the agent’s actions.

The core security assumption is:

> Even if the LLM is manipulated, the system can limit damage if tool calls are validated by an external policy layer.

This follows a defense-in-depth mindset. Prompt instructions may still be useful, but they should not be the only security barrier. Tool execution should be controlled through explicit, auditable, and enforceable policies.

## 1.5 General Objective

To evaluate and propose a least-privilege policy-based approach for mitigating prompt injection attacks in tool-using LLM agents.

## 1.6 Specific Objectives

1. Characterize the problem of prompt injection in tool-using LLM agents.
    
2. Define a taxonomy of tools according to risk, action type, and authorization requirements.
    
3. Implement or adapt an experimental environment for agents with tool calls.
    
4. Create controlled prompt injection scenarios that attempt to induce unauthorized tool calls.
    
5. Implement an external tool-control layer based on security policies.
    
6. Compare different defense strategies: no defense, prompt-only defense, static allowlist, intent validation, risk validation, and combined policy.
    
7. Measure the impact of defenses on both security and utility.
    
8. Produce reproducible artifacts, including code, policies, test scenarios, prompts, metrics, and results.

# 2. Theoretical Background

## 2.1 Large Language Models

Large Language Models are models capable of processing and generating natural language. In traditional applications, they are mainly used to answer questions, summarize texts, classify content, or generate code. In agentic systems, however, the model may also participate in operational decisions, such as choosing tools, planning actions, interpreting results, and deciding next steps.

## 2.2 LLM-Based Agents

An LLM-based agent can be understood as a system that combines:

```text
User input
    ↓
LLM reasoning
    ↓
Action planning
    ↓
Tool selection
    ↓
Tool execution
    ↓
Result interpretation
    ↓
Final answer or next action
```

The difference between a regular chatbot and an agent is the ability to act in an environment. This action may be simple, such as querying a database, or sensitive, such as sending a message, modifying a record, or calling an internal API.

## 2.3 Tool Calling

Tool calling is the mechanism through which an LLM or agent requests the execution of an external function. A tool may be an API, a database query, a file search function, a calendar integration, an HTTP request, or any function exposed by the application.

From a security perspective, each tool has a different level of risk. A tool that only reads public information is less sensitive than one that sends emails, modifies permissions, writes to a database, or accesses confidential data.

## 2.4 Prompt Injection

Prompt injection occurs when textual input manipulates the expected behavior of a model. It can be direct, when the malicious instruction is provided by the user, or indirect, when the malicious instruction is embedded in external content processed by the system.

In tool-using agents, indirect prompt injection is especially important because the model may process content from emails, websites, documents, tool results, or API responses. If these contents contain adversarial instructions, the agent may treat them as commands.

## 2.5 Prompt Injection in Agents

In agents, the impact of prompt injection is greater because the model does not only generate text; it can also influence actions. External content may induce the agent to choose a tool, modify arguments, ignore restrictions, or execute an action outside the original scope of the task.

Therefore, the risk is not limited to the final answer. The entire execution path of the agent becomes part of the security problem.

## 2.6 Excessive Agency

Excessive agency is the risk associated with LLM systems that receive too much autonomy or too many permissions. When a system can call functions, access data, or modify external states without proper control, a reasoning failure or adversarial manipulation can produce more serious consequences.

This research addresses excessive agency by limiting what the agent can execute through external policies.

## 2.7 Least Privilege

The principle of least privilege states that a component should receive only the permissions necessary to perform its task. Applied to LLM agents, this means that an agent should not have unrestricted access to all tools at all times.

Instead, permissions should be granted according to criteria such as:

|Criterion|Example|
|---|---|
|User intent|Does the requested task actually require this tool?|
|Action type|Is the tool read-only or state-changing?|
|Argument scope|Are the parameters within the allowed range?|
|Data sensitivity|Does the tool access private or public data?|
|Reversibility|Can the action be undone?|
|Confirmation need|Does the action require explicit user approval?|

## 2.8 Tool Poisoning

Tool poisoning is a relevant threat in tool-using agent systems. It occurs when tool descriptions, metadata, or outputs contain malicious instructions that influence the model’s behavior. For example, a tool response may include text that attempts to override the agent’s original task or convince the model to call another tool.

This is a form of indirect prompt injection because the malicious instruction reaches the model through external content rather than through the user’s direct request.

## 2.9 Defense in Depth

Defense in depth means combining multiple layers of protection. In LLM agents, this may include prompt design, input validation, isolation of untrusted content, tool-control policies, runtime monitoring, audit logs, and human confirmation for sensitive actions.

This research focuses specifically on the architectural layer responsible for controlling tool execution.

# 3. Related Works

## 3.1 OWASP LLM01: Prompt Injection

OWASP defines prompt injection as a vulnerability in which inputs manipulate the model’s behavior or output. This includes both direct and indirect attacks. Indirect prompt injection is especially relevant to agents because the model may process external content such as documents, emails, websites, or tool outputs.

This work uses OWASP as a practical reference for defining the threat model and motivating the importance of the problem.

## 3.2 OWASP LLM06: Excessive Agency

Excessive agency refers to the risk of LLM systems receiving too much autonomy, functionality, or permission to act. This is directly related to tool-using agents, where the model may be connected to functions capable of reading, writing, sending, modifying, or triggering external operations.

This research can be understood as an architectural response to excessive agency: the agent should not execute every tool call it proposes.

## 3.3 AgentDojo

AgentDojo is an evaluation environment for LLM agents that execute tools over untrusted data. It includes realistic tasks and security test cases designed to evaluate both task utility and robustness against prompt injection attacks.

This research may use AgentDojo as methodological inspiration or as a partial experimental basis, especially because it separates legitimate task success from adversarial success.

## 3.4 MCP Security and Tool Poisoning

The Model Context Protocol provides a standardized way to connect LLM applications to tools and external data sources. However, tool-based architectures introduce risks when tool metadata or outputs include malicious instructions.

Tool poisoning shows that security problems may arise not only from user prompts, but also from the ecosystem of tools exposed to the model. This reinforces the need for external authorization policies.

## 3.5 Defense Against Indirect Prompt Injection

Recent security guidance recommends that systems assume indirect prompt injection may occur and apply layered defenses. These defenses may include isolating untrusted content, monitoring tool use, validating actions, and enforcing policies.

This research follows that direction by proposing a concrete policy layer between the LLM and tool execution.

## 3.6 NIST AI Risk Management Framework

The NIST AI Risk Management Framework provides a broader perspective on managing AI-related risks across design, development, deployment, and evaluation. Although it is not specific to prompt injection, it supports the idea that AI security should be treated as a system-level risk management problem.

This research aligns with that perspective by treating agent security as an architectural and governance issue, not merely a prompt engineering issue.

## 3.7 Literature Review Strategy

The literature review will initially be narrative. The selection criteria will include:

1. works on prompt injection;
    
2. works on LLM agents;
    
3. works on tool calling;
    
4. works on excessive agency;
    
5. works on MCP and tool poisoning;
    
6. works on architectural defenses;
    
7. agent evaluation benchmarks;
    
8. works on authorization, governance, and AI risk management.

## 4.3 Phase 2 — Tool and Risk Taxonomy

A tool taxonomy will be defined for LLM agents. The classification will be based on action type, sensitivity, and risk.

|Category|Description|Expected Risk|
|---|---|---|
|Public read|Queries public or non-sensitive data|Low|
|Internal read|Queries internal or private data|Medium|
|Local write|Changes local files, records, or states|Medium/High|
|Communication|Sends messages, emails, or notifications|High|
|Transactional|Executes purchases, bookings, payments, or critical changes|High|
|Administrative|Changes permissions, users, or configurations|Very High|
|Untrusted external|Queries third-party or unknown external sources|Variable/High|

This taxonomy will serve as the basis for authorization policies.

## 4.4 Phase 3 — Experimental Agent Environment

An agent environment with simulated tools will be implemented or adapted. The environment does not need to execute real-world actions; it may simulate tools and log proposed tool calls.

The baseline architecture will be:

```text
User request
    ↓
LLM agent
    ↓
Plan or tool decision
    ↓
Proposed tool call
    ↓
Policy layer
    ↓
Allow / deny / require confirmation / sanitize arguments
    ↓
Simulated or controlled tool execution
    ↓
Final response
```

Possible simulated tools:

|Tool|Type|Example of Legitimate Use|
|---|---|---|
|search_documents|Read|Search information in documents|
|read_email|Internal read|Retrieve emails related to a task|
|send_email|Communication|Prepare or send a message|
|calendar_lookup|Read|Check availability|
|calendar_create|Write|Create an event|
|update_record|Write|Update a fictitious record|
|web_search|External|Search public information|
|internal_api|Internal|Query a fictitious enterprise system|

Simulated tools are preferable because they allow security evaluation without risk to real systems.

## 4.5 Phase 4 — Prompt Injection Scenarios

Controlled scenarios will be created in which the agent receives external information containing adversarial instructions. These instructions will not be used to attack real systems; they will only evaluate whether the agent attempts to violate tool policies.

Scenario categories:

|Category|Description|
|---|---|
|Forbidden tool induction|External content tries to make the agent use an out-of-scope tool.|
|Argument manipulation|External content tries to modify the parameters of a legitimate call.|
|Tool escalation|External content tries to replace a read tool with a write tool.|
|Improper communication|External content tries to induce message sending without authorization.|
|Simulated exfiltration|External content tries to make the agent access and share fictitious sensitive data.|
|Authority confusion|External content pretends to be a higher-priority system or admin instruction.|
|Intent deviation|External content tries to make the agent pursue a different goal from the original request.|

All data used in the experiments will be fictitious or synthetic.

## 4.6 Phase 5 — Defense Strategies

Six defensive configurations will be compared.

|Code|Defense|Description|
|---|---|---|
|D0|No defense|The agent decides and executes tools without an external layer.|
|D1|Prompt-only defense|The agent is instructed to ignore external commands and use tools carefully.|
|D2|Static allowlist|Each task receives a fixed list of permitted tools.|
|D3|Intent validation|The policy checks whether the proposed tool matches the original user request.|
|D4|Risk validation|Tools are classified by risk; sensitive actions require confirmation or are blocked.|
|D5|Combined policy|Combines allowlist, intent validation, risk classification, and argument validation.|

## 4.7 Agent Tool Policy Guard

The main proposed defense will be called:

# Agent Tool Policy Guard

It will operate as an intermediate layer between the agent and the tools.

```text
LLM Agent
    ↓
Proposed tool call
    ↓
Agent Tool Policy Guard
    ↓
Permission check
    ↓
Intent check
    ↓
Risk check
    ↓
Argument validation
    ↓
Decision: allow, deny, request confirmation, or sanitize
```

## 4.8 Policy Rules

Policies will be described in a declarative format.

Example:

```text
Task: check calendar availability

Allowed tools:
- calendar_lookup

Forbidden tools:
- send_email
- calendar_create
- update_record

Condition:
- calendar_create may only be used if the user explicitly asks to create an event.
```

Another example:

```text
Task: summarize documents

Allowed tools:
- search_documents
- read_document

Forbidden tools:
- send_email
- update_record
- external_post

Condition:
- Retrieved data cannot authorize new tools by itself.
```

## 4.9 Intent Validation

Intent validation checks whether the tool call is related to the original user request.

|User Request|Proposed Tool|Decision|
|---|---|---|
|“Summarize this document.”|read_document|Allow|
|“Summarize this document.”|send_email|Deny|
|“Check my availability tomorrow.”|calendar_lookup|Allow|
|“Check my availability tomorrow.”|calendar_create|Deny or request confirmation|
|“Draft an email to João.”|send_email immediately|Require confirmation|

## 4.10 Risk Validation

Each tool will receive a risk level.

|Level|Tool Type|Policy|
|---|---|---|
|R0|No external effect|Allow if aligned with task|
|R1|Public read|Allow with logging|
|R2|Internal read|Allow only if necessary|
|R3|Write or state-changing action|Require explicit intent|
|R4|External communication|Require confirmation|
|R5|Critical action|Block by default or require strong authorization|

## 4.11 Argument Validation

In addition to allowing or blocking tools, the policy will verify tool arguments.

|Tool|Sensitive Argument|Validation|
|---|---|---|
|send_email|recipient|Must be mentioned by the user or confirmed|
|update_record|modified field|Must be within task scope|
|read_file|file path|Must be inside an allowed directory|
|web_request|URL|Must belong to an allowed or classified domain|
|calendar_create|date and guests|Must match the user’s request|

## 4.14 Formulas

```text
Task Success Rate =
Correctly completed tasks / Total tasks

Unauthorized Tool Call Rate =
Unauthorized proposed calls / Total proposed calls

Unsafe Execution Rate =
Unsafe executed calls / Total unsafe proposed calls

Attack Success Rate =
Successful attacks / Total attack cases

Overblocking Rate =
Legitimate blocked calls / Total legitimate calls

Utility Loss =
Task Success Rate without defense - Task Success Rate with defense
```

# 5. Proposed Solution

## 5.1 Overview

The proposed solution is the **Agent Tool Policy Guard**, an external layer responsible for validating tool calls before execution.

The main premise is:

> The LLM may propose an action, but it should not have final authority to execute it.

## 5.2 Solution Architecture

```text
User
  ↓
LLM Agent
  ↓
Proposed tool call
  ↓
Agent Tool Policy Guard
  ↓
┌──────────────────────────────┐
│ 1. Tool permission check     │
│ 2. Intent validation         │
│ 3. Risk validation           │
│ 4. Argument validation       │
│ 5. Authorization decision    │
└──────────────────────────────┘
  ↓
Tool executed or blocked
  ↓
Final response
```

## 5.3 Components

|Component|Function|
|---|---|
|Tool Registry|Registers available tools and their risk levels.|
|Intent Extractor|Extracts the original user intent.|
|Policy Engine|Applies authorization rules.|
|Argument Validator|Checks whether arguments are within the allowed scope.|
|Risk Classifier|Classifies the proposed action by impact.|
|Confirmation Manager|Requires confirmation for sensitive actions.|
|Audit Logger|Records decisions, blocks, and permissions.|

## 5.4 Possible Decisions

|Decision|Description|
|---|---|
|Allow|The call is permitted.|
|Deny|The call is blocked.|
|Ask Confirmation|The user must explicitly confirm the action.|
|Sanitize Arguments|The call is allowed after argument adjustment.|
|Downgrade Tool|A lower-risk tool replaces the proposed one.|

## 5.5 Example

User request:

```text
“Check whether I have a meeting tomorrow afternoon.”
```

Tool proposed by the agent:

```text
calendar_create(...)
```

Policy decision:

```text
Deny, because the user only requested availability checking, not event creation.
```

Allowed tool:

```text
calendar_lookup(...)
```

## 5.6 Main Difference from Prompt-Only Defenses

|Traditional Defense|Proposed Defense|
|---|---|
|“Instruct the model better”|“Control what the model can execute”|
|Prompt as the main barrier|External policy as the main barrier|
|Model decides tool and authorization|Model proposes, policy authorizes|
|Implicit security|Explicit and auditable security|
|Hard to verify|Rules and logs can be inspected|

This approach does not eliminate the need for good prompts, but it reduces dependence on prompts as the only line of defense.

## Annex D — Recommended Final Scope

The recommended scope for this research is:

> To evaluate whether an external least-privilege policy layer reduces unauthorized tool calls in LLM agents under prompt injection attacks, comparing prompt-only defense, allowlists, intent validation, risk validation, and combined policy, while measuring both security and utility.

The central thesis is:

> **The security of LLM agents should not depend only on the model’s obedience to prompts; it should be reinforced through external, auditable, least-privilege policies.**
